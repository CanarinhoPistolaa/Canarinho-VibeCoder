## Codebase Patterns

- Workflow validation tests auto-discover all workflow directories and validate:
  - YAML parsing (id matches directory name)
  - All workspace.files paths resolve to existing files
  - Agent roles are valid (analysis, coding, verification, testing, pr, scanning)
  - Description is non-empty
  - All agents declare tamandua-agents skill
  - All steps reference valid agents
  - Polling config is valid
- README catalog test (`readme-workflow-catalog.test.ts`) auto-discovers workflows and checks README has entries for each
- Shared setup personas live at `agents/shared/setup/` and are referenced via relative paths (`../../agents/shared/setup/AGENTS.md`) from workflow agent definitions
- Merger agent persona files are workflow-local (not shared) because merge instructions vary per workflow family
- Merge workflows use `role: pr` for merger agent; non-merge workflows have no merger
- Shell-script shebang `#!/bin/sh` requires `bash <script>` or `sh <script>` invocation (not direct `node <script>`)
- The Tamandua CLI dist is at `dist/cli/cli.js` and the shell wrapper at `bin/tamandua` resolves it
- commit convention: `Co-Authored-By: Tamandua <tamandua@tetradactyla.org>` footer required on every commit
- Worktree variant agent symlink chains:
  - Worktree variant agents/ symlink to merge variant agents/
  - Merge variant setup agent symlinks to shared setup (`../../../agents/shared/setup`)
  - The chain: merge-worktree/agents/X -> merge/agents/X -> (for setup) shared/setup
  - Symlink depth is important: from merge/agents/, shared setup requires 3 levels up (`../../../agents/shared/setup`) because agents/ is inside the workflow directory (`workflows/<id>/agents/`)
- Worktree workflow.yml changes from merge variant:
  - `id` changes to add `-worktree` suffix
  - `run.workspace: worktree` is added
  - Setup step gets `WORKTREE_ORIGIN_REPOSITORY` and note that `ORIGINAL_BRANCH` is pre-seeded
  - Setup step must NOT include `git branch --show-current` command
  - Finalize_merge step gets `WORKTREE_ORIGIN_REPOSITORY` and "Worktree Mode" instructions with `git -C`
  - All Phase 1 and Phase 3 cd/git commands use `{{worktree_origin_repository}}`
  - Phase 2 rebase still uses `{{repo}}` (the worktree)
- README table variant naming convention: Local-only, + Merge, Worktree, Worktree + Merge (never "+ Merge (Worktree)")
