## Codebase Patterns
- This project uses node:sqlite with sync API (DatabaseSync, not async)
- All step-ops functions are in src/installer/step-ops.ts
- Tests use node:test, run with `node --test`
- KEY: value output parsing uses `parseOutputKeyValues` which lowercases keys
- Template resolution uses `resolveTemplate` with {{key}} placeholders
- Missing template key detection uses `findMissingTemplateKeys`
- Workflow YAML files have "Reply with:" sections describing expected agent output
- Reply-with blocks have uppercase KEY: lines like `STATUS:`, `BRANCH:`, `CHANGES:`, `TESTS:`
- Step input templates are stored in the steps table's `input_template` column as formatted strings with indentation from the workflow YAML
- `parseExpectedKeys` extracts expected keys from Reply-with blocks and returns them lowercased (consistent with `parseOutputKeyValues` convention)
- `findProducerForMissingKey` uses `parseExpectedKeys` internally to check which keys upstream steps declare
- Test infrastructure pattern: isolate with temp HOME+TAMANDUA_STATE_DIR+TAMANDUA_DB_PATH, seed runs+steps via direct SQL, call function, assert
- `resolveStepContext` collects context from run's context JSON + previous DONE steps' KEY:value outputs
- claimStep populates special keys (retry_feedback, verify_feedback, timeout_retry) before template resolution, so they are never treated as missing
- The "quarantine-broken-tests" family of workflows have setup as the first step (no planner), so `branch` must come from the run's seeded context — the simulation needs to include it
- Events are emitted to a JSONL file; `getRunEvents(runId)` reads them back filtered; `emitRunTerminalEvent` is a private helper that includes `tokensSpent` in terminal run events
- Loop claim path tests require: (a) stories inserted into the stories table, (b) a loop step with loop_config={"over":"stories"}, (c) seedStories helper that inserts a pending US-001 story
- Loop path missing-key blocking unclaims both the story and the loop step; both must be reset to pending
- Story context keys (current_story, current_story_id, current_story_title, completed_stories, stories_remaining, progress) are populated AFTER story claim but BEFORE findMissingTemplateKeys — they only appear in resolvedInput if the template explicitly references them

## 2026-07-03 - US-002: Add findProducerForMissingKey helper to locate the upstream step responsible for a missing template key
- Added `ProducerResult` interface to `src/installer/step-ops.ts`:
  - `{ stepId: string; stepIndex: number; retryCount: number; maxRetries: number }`
- Added `findProducerForMissingKey(runId, currentStepIndex, missingKey)` function
- Queries upstream steps where `step_index < currentStepIndex AND status = 'done'` ordered by `step_index DESC`
- For each upstream DONE step, calls `parseExpectedKeys` on its `input_template`
- Returns the first (most recent) step whose expected keys include the missing key
- Returns `null` when no upstream DONE step declares the missing key
- Case-insensitive key matching: both search key and parsed keys are lowercased
- Added 8 unit tests:
  - Returns most recent upstream DONE step declaring the key
  - Returns null when no upstream step declares the key
  - Skips non-done upstream steps (running, waiting, pending, failed)
  - Returns correct retry counts (retryCount, maxRetries)
  - Returns null when no upstream steps exist at all
  - Returns null when upstream done steps exist but none declare the key
  - Case-insensitive matching for the lookup key
  - Most-recent-wins when multiple upstream steps declare the same key
- Files changed:
  - `src/installer/step-ops.ts`: +54 lines (interface + function)
  - `tests/step-ops.test.ts`: +235 lines (8 tests + describe block)
- **Learnings:** The steps table has `retry_count` and `max_retries` columns (note snake_case in DB). `parseExpectedKeys` already lowercases all parsed keys, so `findProducerForMissingKey` also lowercases the search key for a consistent comparison. The function returns the DB row `id` as `stepId` (matching how `claimStep` uses it in the `ClaimResult` shape).

## 2026-07-03 - US-003: Refactor claimStep to block unresolvable templates and route recovery instead of substituting [missing: key]
- Replaced the two "warn and substitute" branches in claimStep (loop story claim and single-step claim) with blocking logic via `resolveMissingKeys()`
- Added `MissingKeyAction` type: `'proceed' | 'rejected' | string`
- Added `resolveMissingKeys(runId, currentStepIndex, consumerStepId, consumerStepRowId, agentId, missingKeys)` function:
  - Phase 1: For each missing key, calls `findProducerForMissingKey`; collects into producer map (deduplicated), unresolvable list, and exhausted list
  - Phase 2: If any key has no producer → fail the run immediately (`failRunForMissingTemplateKeys`)
  - Phase 3: If any producer's retries are exhausted → fail the run immediately
  - Phase 4: Re-pend all producers with `retry_feedback` naming the missing keys, increment `retry_count`, emit `step.repended` event
  - Returns `'rejected'` on re-pend, or a failure message string on fail-fast
- Added `failRunForMissingTemplateKeys(stepRowId, stepId, runId, agentId, message)`: marks step + run as failed, emits `step.failed` + `run.failed` events, tears down crons
- In claimStep's loop claim path: after populating special keys (`retry_feedback`, `verify_feedback`, `timeout_retry`), calls `resolveMissingKeys`. On `'rejected'`: unclaims story and loop step. On fail: bail.
- In claimStep's single step claim path: same pattern — calls `resolveMissingKeys` after populating special keys; on `'rejected'`, unclaims the step.
- Special keys (`retry_feedback`, `verify_feedback`, `timeout_retry`) are populated before template resolution, so they are never flagged as missing
- When no keys are missing: `resolveMissingKeys` returns `'proceed'` and claimStep works exactly as before
- Simulation fix: added `branch` to seeded context in `tests/workflow-graph-simulation.test.ts` — quarantine workflows have setup as first step (no planner), so `branch` must come from seed context
- Added 10 comprehensive unit tests covering all acceptance criteria:
  1. Producer re-pend: claimStep blocks, producer re-pended with retry_feedback, consumer unclaimed
  2. No-producer fail-fast: step + run marked failed with message naming unresolvable key
  3. Exhausted producer fail-fast: run fails when producer at retry_count >= max_retries
  4. Normal path: claimStep resolves templates normally when no keys missing
  5. Special keys: retry_feedback, verify_feedback, timeout_retry never flagged as missing
  6. step.repended event emitted on re-pend path
  7. step.failed + run.failed events emitted on fail-fast path
  8. Boundedness: producer re-pend that itself still fails eventually exhausts retries (no infinite ping-pong)
  9. Multiple missing keys from same producer: producer re-pended once, feedback names all missing keys
  10. Mixed resolvable+unresolvable keys: unresolvable takes priority → fail-fast (no partial re-pend)
- Files changed:
  - `src/installer/step-ops.ts`: +201 lines (resolveMissingKeys, failRunForMissingTemplateKeys, claimStep modifications)
  - `tests/step-ops.test.ts`: +428 lines (10 tests + describe block, getRunEvents import)
  - `tests/workflow-graph-simulation.test.ts`: +1 line (branch in seeded context)
- **Learnings:** claimStep populates `retry_feedback`, `verify_feedback`, and `timeout_retry` before calling `findMissingTemplateKeys`, so those keys are always present. The `resolveMissingKeys` function deduplicates by producer stepId — multiple missing keys from the same producer only re-pend that producer once. `getRunEvents` from the events module reads JSONL events filtered by runId — usable in tests to verify event emission.
