## Codebase Patterns

- **Registration failure handling:** When `registerRunWithDaemon()` returns null/non-2xx after run creation, use `scheduleRunCronTeardown(runId)` for fire-and-forget scheduling teardown (calls `removeRunCrons` + `terminateRunWithDaemon` via dynamic import).
- **Terminal run marking:** Set `status = 'failed'` and `scheduling_status = NULL` in a single UPDATE; the reconciler filters `WHERE status IN ('running')` so terminal runs are automatically excluded.
- **Event emission:** Use `emitEvent()` from `./events.js` — writes to run-specific and global JSONL files. The event should use `detail` field for failure context.
- **Tests use self-contained pattern:** In-memory `DatabaseSync`, custom `test()`/`assert()` functions, no framework dependencies. Tests import from `dist/` via dynamic `import()` for ESM compatibility.
- **`scheduleRunCronTeardown`** is exported from `./step-ops.js` and handles DB update (clears scheduling_status for terminal runs), cron removal, and daemon termination — all fire-and-forget.
- **Reconciler query** in `control-server.ts`: `WHERE status IN ('running') AND (scheduling_status IS NULL OR scheduling_status IN ('pending_register', 'active', 'error'))`
- **Resume registration failure pattern** mirrors new-run registration failure: same 3-line pattern (UPDATE status='failed' + scheduling_status=NULL, emitEvent, scheduleRunCronTeardown), with event detail prefixed "Resume registration failed:" instead of "Registration failed:"
- **just-do-it dispatcher workspace:** When `workflowId === "just-do-it"` and workspace mode is direct, the harness working directory is redirected to a neutral per-run workspace under `<tamandua-state>/just-do-it-workspaces/<runId>`. The original target repo is preserved in `target_working_directory_for_harness` context key. The `repo` context key points to the target repo (not the neutral workspace) for git operations.
- **resolvePiStateDir()** from `./paths.js` returns `process.env.TAMANDUA_STATE_DIR` or `~/.tamandua`. Imported alongside `resolveWorkflowDir`.
- **Workflow validation test extraction:** Dispatch step input is extracted from workflow.yml by splitting on `"input: |"` and `"\n    expects:"`. The `dispatchStepInput` variable is used for targeted tests.
- **Regex patterns for CLI flag validation:** Use `--flag\s+"` to match flag-as-argument (not advisory text). Use negative lookahead `--no-hurry(?!-please-save-tokens-mode)` for bare flag detection where no advisory mentions exist. For multi-line matching in `assert.match`, check terms individually rather than using `.*` across newlines.
