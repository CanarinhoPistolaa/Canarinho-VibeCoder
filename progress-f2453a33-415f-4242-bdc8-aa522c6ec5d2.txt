## Codebase Patterns

- `src/installer/run-cleanup.ts` contains `/proc` scanning helpers and `sweepRunProcesses` main export
- Tests live alongside source: `src/installer/run-cleanup.test.ts`
- Build with `npm run build` (tsc + HTML copy + version injection)
- Test with `npm test` which runs `node --test tests/*.test.ts src/**/*.test.ts`
- ESM project тАФ dist files use `export` keyword, not CJS `exports.`
- `getRunWorktree(runId)` in `src/installer/worktree-manager.ts` retrieves worktree metadata from DB
- `removeRunCrons` in `src/installer/agent-scheduler.ts` tears down dispatch jobs for a run
- Module-level Maps in agent-scheduler.ts are the pattern for in-memory state (activeTimers, pendingStartTimers, jobMetadata, inFlightChildren)
- Internal helpers exposed for tests use `_` prefix (e.g., `_scheduledJobCountForRun`, `_pendingSweepTimerCount`)
- `shutdownAllCrons()` is the teardown hook for test isolation тАФ must clear all Maps
- The daemon-resident sweep timer pattern: schedule an unref-ed timer in removeRunCrons, dedup via Map<runId, Timeout>, import runtime deps (worktree-manager, run-cleanup) dynamically within the timer callback
- Doctor check functions follow the `run<Category>Checks(opts?: DoctorOpts)` pattern with DB path isolation via save/restore of canarinho_DB_PATH
- Doctor tests seed DBs manually (DatabaseSync + SQL) for fully isolated scenarios; temp HOME per test
- Process-matching helpers (getProcPids, processBelongsToRun) are now exported from run-cleanup.ts and importable by doctor.ts
- MOTOR-CONTRACT.md sections: Lifecycle & pipeline, Dispatch, Deterministic-motor guarantees, Failure & recovery, Control plane & scheduling lifecycle, Token accounting, Workspace
