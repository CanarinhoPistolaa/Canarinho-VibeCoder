## Codebase Patterns
- npm test sets TAMANDUA_PI_BINARY=/bin/false as a safety net — this env var passes through cleanChildEnv
- Unit tests that spawn pi through runPiAgent manipulate PATH, not TAMANDUA_PI_BINARY
- The agent-scheduler's findPiBinary checks TAMANDUA_PI_BINARY first (with executable check), then falls back to PATH
- autoloresearch tests run in-process (no daemon), sharing process.env with other test files
- Use beforeEach/afterEach hooks to save/restore TAMANDUA_PI_BINARY when tests manipulate PATH
- e2e tests for autoresearch use CLI spawn (not in-process — unlike unit tests). Isolate with temp HOME + TAMANDUA_STATE_DIR + TAMANDUA_DB_PATH, don't need a daemon.
- Fake pi scripts for autoresearch e2e: write change to working tree + emit pi JSON with message_end (assistant role) containing STATUS: done/CHANGES/HYPOTHESIS/LEARNED/NEXT_FOCUS
- Baseline runs (no --prompt) have committed=false because no source files change (only autoresearch protected files are created)
- Autoresearch protected files (autoresearch.config.json, autoresearch.jsonl, autoresearch.md, autoresearch.sh) are intentionally excluded from commits and show as untracked (??) in git status
- RULE: If npm test AND ./run-all-e2e-tests both pass, a change is safe — npm test guards structural correctness (unit + integration) while run-all-e2e-tests guards the real daemon/scheduler/harness/CLI paths against regressions in auth isolation, worktree plumbing, and pipeline state machines
