## Codebase Patterns

- `agent-scheduler.ts` uses module-level state (`inFlightJobs`, `jobMetadata`, `activeTimers`, `pendingStartTimers`, `inFlightChildren`) — no class, just const maps/sets
- Tests import from `../../dist/...` and run with `node --test src/**/*.test.ts` after `npm run build` (tsc)
- Test helpers use isolated state via `shutdownAllCrons()` in `afterEach`
- `setupAgentCrons` takes `workflow`, `runId`, `options` and creates `CreateCronJobParams`
- `executePollingRound` is the core polling function called by setInterval timers
- `inFlightJobs` Set is used to prevent duplicate concurrent pi processes for the same job
- Workflow specs are loaded from disk via `loadWorkflowSpec(resolveWorkflowDir(workflowId))`
- Agent IDs in `jobMetadata` are stored as full prefixed form `workflowId_shortAgent`
- For nudge tests, use temp HOME + TAMANDUA_STATE_DIR to isolate workflow filesystem access
- Control-server handlers like `handleRegisterRun`, `handlePauseRun` use `import()` for dynamic imports to avoid circular deps
- DB direct insertion in tests uses `import("node:sqlite")` and the daemon's temp db path
- `handleRegisterRun` is idempotent — calling it on an already-admitted run returns `{ state: "active" }`
- The SQL `WHERE status = 'running'` naturally excludes both paused and terminal runs
- CLI tests use `cli()` helper that spawns `child_process.spawnSync` with temp HOME and TAMANDUA_STATE_DIR
- Daemon-dependent CLI tests spawn `node dist/server/daemon.js <port>` with `cleanChildEnv`, wait for `/control/health` to respond 200
- `logs-tail-format.ts` exports `formatLogsTailLabel`, `formatLogsTailLine`, `formatLogsTailLines` and uses a central `EVENT_LABELS` Record for friendly display names
- Event label tests use `makeEvent()` helper that constructs minimal `TamanduaEvent` objects, imported from `../../dist/installer/events.js`

- Added `tryMarkJobInFlight(jobId: string): boolean` to `src/installer/agent-scheduler.ts`
  - Atomically checks `inFlightJobs` and adds if not present
  - Synchronous (no await) to close TOCTOU window

  - Was: after stale-claim sweeper, before persona reading
  - Now: immediately after working-directory check, before run-status check
- Preserved existing `finally` block that calls `inFlightJobs.delete(job.id)`
- Added 7 tests in `src/installer/agent-scheduler.test.ts` covering:
  - first call returns true
  - second call returns false
  - different jobIds are independent
  - three-call sequence
  - concurrent-call simulation

- **Learnings:** The key race was that `inFlightJobs` check happened *after* several awaited calls (run status DB lookup, stale-claim recovery). Moving the check-and-add before any `await` eliminates the window where two concurrent calls could both pass the guard.

## 2026-05-24 - US-003: Add POST /control/nudge endpoint and handler to control-server
- Added `handleNudge()` function to `src/server/control-server.ts`:
  - Queries SQLite for runs with `status = 'running'` (excludes paused and terminal)
  - Calls `handleRegisterRun()` for each running run to attempt admission (idempotent)
  - Collects successfully admitted run IDs (skip paused/queued/non-active)
  - Calls `nudgeScheduledRuns()` with the collected run IDs
  - Emits `run.nudged` events for each run with launched/skipped counts
  - Emits `agent.nudged` and `agent.nudge.skipped` events per job
  - Returns aggregate JSON response: `{ runningRuns, scheduledRuns, launched, skippedInFlight, skippedPaused, runs, errors }`
  - Returns zero counts cleanly when no runs are running
- Added `POST /control/nudge` route in `createControlServer()` with x-tamandua-secret auth
- Updated file header comment to document the new endpoint
- Added 6 tests in `src/server/control-server.test.ts`:
  - returns zero counts when no runs are running
  - requires valid x-tamandua-secret (401 if missing)
  - excludes paused runs
  - excludes terminal runs (completed, failed, canceled)
  - returns aggregate counts when runs are running
  - emits events with correct shape
- Files changed: `src/server/control-server.ts`, `src/server/control-server.test.ts`
- **Learnings:**
  - `handleRegisterRun()` is idempotent — calling it on an already-active run returns `{ state: "active" }`
  - Paused runs (status='paused') and terminal runs (completed/failed/canceled) are naturally excluded by `WHERE status = 'running'`
  - Nudge events (`run.nudged`, `agent.nudged`, `agent.nudge.skipped`) follow the same `emitEvent()` pattern as existing events (`run.paused`, `run.resumed`)
  - The response includes per-run detail (`runs[]`) for diagnostics, matching the design doc shape

## 2026-05-24 - US-005: Add tamandua nudge CLI command
- Added `getNudgeHelp()` function to `src/cli/cli.ts` with usage and description
- Added `nudge` to global help in `getUsageText()`
- Added `tamandua nudge` to the `--help` dispatch
- Added `nudge` to `shouldSkipUpdateWarning()` (suppress update warning noise)
- Added `nudge` command handler in `main()`:
  1. Rejects extra arguments with clear usage message
  2. Calls `nudgeWithDaemon()` 
  3. If daemon unreachable: tries `ensureDaemonControlAvailable()`, retries nudge
  4. If still unreachable: prints "Failed to nudge: control plane is not reachable." and exits 1
  5. If non-200 response: prints error message and exits 1
  6. If zero running runs: prints "No running Tamandua runs to nudge."
  7. If successful: prints summary "Nudged {N} running run(s): launched {L} agent(s), skipped {S} in-flight."
- Added 6 tests in `src/cli/cli.test.ts`:
  - tamandua --help includes tamandua nudge in command listing
  - tamandua nudge --help shows usage with no args/no options
  - tamandua nudge extra-arg fails with usage message
  - tamandua nudge with daemon unreachable prints clear error message
  - tamandua nudge with daemon running and no runs prints zero-runs message
  - tamandua nudge with daemon running and active runs prints summary
- Files changed: `src/cli/cli.ts`, `src/cli/cli.test.ts`
- **Learnings:**
  - The `ensureDaemonControlAvailable()` function starts the daemon if needed and waits for the control plane to become reachable
  - Daemon-dependent CLI tests need to spawn the daemon with `cleanChildEnv`, wait for `/control/health` to return 200, and then run the CLI with matching HOME/TAMANDUA_CONTROL_PORT env vars
  - The daemon's reconciler lazily creates the DB schema on its first tick — tests inserting runs must `await sleep(2000)` before touching the DB
  - `__dirname` is not available in ESM test files — must use `fileURLToPath(import.meta.url)` + `path.dirname()` instead
