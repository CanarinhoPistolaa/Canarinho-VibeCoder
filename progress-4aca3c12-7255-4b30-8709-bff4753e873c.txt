## Codebase Patterns

- Tests use Node's built-in `node:test` and `node:assert/strict`. No external test runners.
- `npm test` runs `node --test tests/*.test.ts src/**/*.test.ts` — tests import from dist/.
- `tsconfig.json` only compiles `src/**/*.ts`, excludes `tests/` and `src/**/*.test.ts`.
- All test scripts (`./build`, `./run-all-tests`, etc.) are bash with `set -euo pipefail` and `cd "$(dirname "$0")"`.
- Test isolation uses `fs.mkdtempSync()` for temp HOME directories with cleanup in `finally` blocks.
- Tests that need the built CLI invoke `tamandua` via shell, not via direct imports.
- Git commits must end with: `Co-Authored-By: Tamandua <tamandua@tetradactyla.org>`
- When running `node --test` from within an existing `node --test` process, must unset `NODE_TEST_CONTEXT` env var to avoid recursive test detection skipping subprocess tests.
- The `tamandua` binary at `bin/tamandua` is a shell script (`#!/bin/sh`), not a JS file. Use `tamandua` on PATH, not `node bin/tamandua`.
- `step claim` requires the FULL UUID run ID, not the 8-char prefix. The 8-char prefix from `workflow run` output must be resolved via DB query.
- `tamandua workflow install` requires `~/.pi/agent/settings.json` to exist (minimal pi config).
- Worktree-mode workflows need `--worktree-origin-repository` pointing to a clean git repo (no uncommitted changes).
- The daemon auto-starts when running `tamandua workflow run`; it spawns detached and must be stopped via `tamandua dashboard stop`.
- Test suites that spawn daemon processes should use `{ concurrency: 1 }` to prevent port conflicts.
- Workflow pipeline step advancement is purely DB-driven; completing steps with `tamandua step complete` advances the pipeline without requiring actual agent execution.
- For loop steps with `verify_each`, completing the implement step triggers the verify step. Completing the verify step calls `checkLoopContinuation` which marks the loop done when all stories are done.
